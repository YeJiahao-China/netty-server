# 协议接入开发指南

本文档面向**外部协议 jar 开发者**。阅读本文档后，你将能够为一个已经部署运行的 netty-server 实例**新增一种 TCP 协议接入能力**，且无需修改服务端任何代码、无需重启服务。

---

## 一、工作原理速览

netty-server 在启动后会监听一组端口，每个端口绑定一种协议。当客户端 TCP 连接进来时，框架会：

1. 取出当前连接所在的服务端口
2. 查询「端口 → 协议名」映射，找到对应的 `ProtocolDecoderProvider`
3. 调用 Provider 的 `createHandlers()` 拿到一组 Netty Handler
4. 按顺序装入 ChannelPipeline

因此，**你要做的事情就是**：实现 `ProtocolDecoderProvider` 接口 → 打成 jar → 通过 HTTP 接口让它被加载并绑定到端口。

---

## 二、添加依赖

你的协议 jar 只需要主项目暴露的 SPI 接口和 netty-all。**这两者都必须用 `provided` scope**，避免把 netty 类重复打包进你的 jar，导致 ClassLoader 冲突。

```xml
<dependencies>
    <!-- netty-all: provided，由主项目运行时提供 -->
    <dependency>
        <groupId>io.netty</groupId>
        <artifactId>netty-all</artifactId>
        <version>4.1.97.Final</version>
        <scope>provided</scope>
    </dependency>

    <!-- 主项目 SPI 接口: provided -->
    <dependency>
        <groupId>com.yjh</groupId>
        <artifactId>netty-server</artifactId>
        <version>1.0</version>
        <scope>provided</scope>
    </dependency>
</dependencies>
```

> 提示：如果暂时无法依赖主项目 jar，可以只把 `ProtocolDecoderProvider.java` 这个接口文件复制到你的项目中（接口稳定后基本不变），同样可以工作。

## 三、实现 Provider 接口

### 3.1 接口定义（摘录）

```java
public interface ProtocolDecoderProvider {
    String name();                                            // 协议唯一标识
    default String version() { return "1.0.0"; }              // 版本
    default String description() { return ""; }               // 描述（展示用）
    ChannelHandler[] createHandlers();                        // 创建 Handler 链
    default IdleConfig idleConfig() { return null; }          // 读写超时配置
    default void destroy() { }                                // 卸载时回调
}
```

### 3.2 示例：Echo 协议

```java
package com.acme.netty.echo;

import com.cas.access.netty.protocol.ProtocolDecoderProvider;
import io.netty.channel.ChannelHandler;
import io.netty.channel.ChannelInboundHandlerAdapter;
import io.netty.channel.ChannelHandlerContext;
import io.netty.handler.codec.string.StringEncoder;

import java.nio.charset.StandardCharsets;

public class EchoProtocolProvider implements ProtocolDecoderProvider {

    @Override
    public String name() {
        return "echo";
    }

    @Override
    public String version() {
        return "1.0.0";
    }

    @Override
    public String description() {
        return "Echo 协议示例：原样回显客户端发送的数据";
    }

    @Override
    public ChannelHandler[] createHandlers() {
        // 每次调用都必须返回新实例！ByteToMessageDecoder 等是有状态的
        return new ChannelHandler[]{
                new EchoHandler(),
                new StringEncoder(StandardCharsets.UTF_8)
        };
    }

    @Override
    public IdleConfig idleConfig() {
        // 客户端 300 秒不发数据就认为超时
        return new IdleConfig(300, 0, 0);
    }

    /** 简单的回显 Handler */
    public static class EchoHandler extends ChannelInboundHandlerAdapter {
        @Override
        public void channelRead(ChannelHandlerContext ctx, Object msg) {
            ctx.writeAndFlush(msg);   // 把读到的数据原样写回
        }
    }
}
```

## 四、SPI 注册文件【关键】

在 jar 中**必须**包含以下文件，否则框架无法发现你的实现：

```
src/main/resources/META-INF/services/com.cas.access.netty.protocol.ProtocolDecoderProvider
```

文件内容（每行一个实现类全限定名）：

```
com.acme.netty.echo.EchoProtocolProvider
```

一个 jar 可以注册多个 Provider（每行一个）。

## 六、运行时管理

服务端提供 HTTP 接口（默认端口 `2310`）：

### 6.1 加载新协议

```bash
curl -X POST http://server-host:2310/protocols/reload
```

返回示例：
```json
{
  "success": true,
  "loaded": ["echo:1.0.0 -> OK"]
}
```

### 6.2 查询当前已加载协议

```bash
curl http://server-host:2310/protocols
```

返回示例：
```json
{
  "protocols": [
    { "name": "hj212", "version": "1.0.0", "source": "builtin", "active": true },
    { "name": "echo",  "version": "1.0.0", "source": "/path/protocol-echo.jar", "active": true }
  ],
  "bindings": { "10101": "hj212" }
}
```

### 6.3 绑定协议到端口

```bash
curl -X POST http://server-host:2310/protocols/echo/bind/10102
```

绑定后客户端即可连接 10102 端口使用 echo 协议。

### 6.4 卸载流程

卸载时框架会**自动**完成全部清理工作，调用方无需先解绑端口：

```bash
curl -X DELETE http://localhost:2310/protocols/echo
```

框架内部依次执行：
1. 关闭该协议绑定的所有端口监听
2. 强制断开这些端口下的所有活跃连接（保证 Pipeline 销毁，避免 ClassLoader 泄漏）
3. 调用 Provider 的 `destroy()` 让实现方清理自身资源
4. 关闭 URLClassLoader，释放 jar 文件句柄（Windows 文件锁解除）

返回示例：
```json
{
  "success": true,
  "closedPorts": [10102],
  "message": "已自动关闭端口监听并断开所有活跃连接: [10102]"
}
```

> ⚠️ 注意：卸载会强制断开所有客户端连接。如果只想替换 jar 但保留连接，
> 使用 `POST /protocols/reload` 触发热替换（旧连接用旧 Provider 继续工作）。

### 6.5 热升级

直接用新版本 jar 覆盖 `protocols/` 下的旧 jar，然后调用 `reload`：

```bash
curl -X POST http://server-host:2310/protocols/reload
```

- 旧连接继续用旧 Handler 运行（Pipeline 中的实例不会被替换）
- 新连接会走新版本 Provider
- 旧连接最终因客户端断开或空闲超时自然消亡

## 七、注意事项（避坑指南）

### 7.1 createHandlers() 必须返回新实例

Netty 的 `ByteToMessageDecoder`、`ChannelInboundHandlerAdapter` 等**默认不是 `@Sharable`**，多个连接复用同一实例会导致状态错乱、Buffer 串读。

```java
// ❌ 错误：所有连接共享同一个 Handler
private final EchoHandler shared = new EchoHandler();
public ChannelHandler[] createHandlers() {
    return new ChannelHandler[]{ shared };
}

// ✅ 正确：每次调用都 new
public ChannelHandler[] createHandlers() {
    return new ChannelHandler[]{ new EchoHandler() };
}
```

如果你的 Handler 是无状态且标注了 `@ChannelHandler.Sharable`，可以共享。

### 7.2 禁用 Spring 注解

外部协议 jar 中的类**不要**标注 `@Component`、`@Service`、`@Configuration` 等注解。这些类由独立的 `URLClassLoader` 加载，不在 Spring 容器中，标注注解也不会生效，反而可能造成混淆。

### 7.3 包名约定

为了避免被主项目的 `@SpringBootApplication` 包扫描误扫（默认扫 `com.cas.access`），**外部协议 jar 的包名不要用 `com.cas.access` 开头**。建议用你自己的域名反向，如 `com.acme.netty.xxx`。

### 7.4 协议名必须全局唯一

`name()` 返回值在同一个 netty-server 实例中必须唯一。建议用反向域名风格：

```java
public String name() {
    return "com.acme.modbus-tcp";  // 或者简单地 "modbus-tcp"
}
```

### 7.5 不要把 netty-all 打进你的 jar

主项目已经有完整的 netty-all 4.1.97，重复打包会导致 `ClassCastException`（同一个类被两个 ClassLoader 加载）。务必在 pom.xml 中用 `<scope>provided</scope>`。

### 7.6 处理 ByteBuf 的引用计数

如果你的 Handler 直接处理 `ByteBuf`，记得遵循 Netty 的引用计数规则：

```java
@Override
public void channelRead(ChannelHandlerContext ctx, Object msg) {
    try {
        ByteBuf buf = (ByteBuf) msg;
        // ... 处理 buf
    } finally {
        ReferenceCountUtil.release(msg);  // 必须释放
    }
}
```

或者把 ByteBuf 转换成 POJO 后再传给下游，避免泄漏。

## 八、调试技巧

### 8.1 看日志

服务端日志会清晰地反映加载过程：

```
协议注册成功: name=echo, version=1.0.0, source=/path/protocol-echo.jar
协议[echo]绑定到端口[10102]
[Pipeline装配成功] client=127.0.0.1:54321 port=10102 protocol=echo
```

### 8.2 测试本地连接

```bash
# 用 nc 或 telnet 模拟客户端
nc localhost 10102
hello
# 应该立即看到回显的 hello
```

### 8.3 排查 ServiceLoader 没找到实现

最常见的原因：
- `META-INF/services/` 文件路径错误（必须在 `src/main/resources/` 下）
- 文件名错误（必须是接口全限定名 `com.cas.access.netty.protocol.ProtocolDecoderProvider`）
- 文件内容有拼写错误（实现类全限定名写错）
- 用 `jar tf your.jar | grep META-INF/services` 检查是否真的打进了 jar

## 十、FAQ

**Q1：我的协议需要读 Redis、调 HTTP 接口怎么办？**

A：Provider 接口的 `destroy()` 方法返回时框架已经清理 ClassLoader。如果你的协议需要外部资源（连接池、客户端等），请在 Provider 内部自己管理生命周期（构造时创建，`destroy()` 时关闭）。注意这些资源也是用独立的 ClassLoader 加载的，不要依赖 Spring 注入。

**Q2：能否在一个 jar 里放多个协议？**

A：可以。`META-INF/services/` 文件中每行写一个 Provider 实现类即可，每个 Provider 有独立的 `name()`。

**Q3：升级版本时旧连接还在用旧 Handler，会不会出问题？**

A：不会。Netty Pipeline 中已经装配的 Handler 实例不受 Provider 替换影响，旧连接继续用旧逻辑。新连接走新版本。如需强制切换所有连接到新版本，调用 `DELETE /protocols/{name}` 卸载协议（框架会自动断开旧连接并关闭端口），然后重新 reload + bind。

**Q4：协议 jar 中的 log 怎么打？**

A：直接用 SLF4J API 即可，主项目已经提供了 logback 实现：

```java
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;

private static final Logger log = LoggerFactory.getLogger(EchoHandler.class);
log.info("...");
```

外部 jar 的日志会与主项目日志合并到同一输出。
